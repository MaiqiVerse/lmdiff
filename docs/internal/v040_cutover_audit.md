# v0.4.0 commit 4.0 — pre-flight cutover audit

> **Status**: audit only — no code changes. Implementation gated on
> review of this document. See spec for tasks 1–7 once approved.

This audit traces every code path that today routes `lmdiff.compare()`
/ `lmdiff.family()` through the v0.2.x `InferenceEngine`, identifies
where the Engine Protocol falls short of replacing it, and recommends
a category for each gap (extend Protocol / inline-rewrite / private
bridge). All file:line references are against `main` at commit
`8444de0` (v0.3.2 + L-029 retrospective).

## TL;DR

- **Scope is well-bounded**: 5 attribute accesses on `InferenceEngine`
  outside the Protocol, all in `lmdiff/geometry.py`. No surprises in
  metric or report layers.
- **The two real Protocol gaps** are (1) batched `score`/`generate`
  signatures, and (2) runtime-only override kwargs (`system_prompt`,
  `context`, `decode`). Both have a clean inline-rewrite path that
  doesn't widen the Protocol.
- **Recommended scope**: full cutover (`compare()` + `family()`) in a
  single commit. `family()` and `compare()` share the same internal
  pipeline (`ChangeGeometry.analyze`); splitting them buys nothing.
- **No new Protocol methods needed**. One small Protocol-conforming
  helper is added to support the BPB cross-tokenizer fallback (option
  A in Audit 4 — minimal surface).
- **Test baseline**: 861 passed / 4 skipped on `not slow and not gpu`.
  Cutover must not regress this number.
- **Calibration baseline**: needs to be generated by you (GPU-gated).
  Snippet at the end of this audit; commit the JSON to
  `tests/fixtures/calibration_v032_baseline.json`.

The current `Engine` Protocol (`lmdiff/_engine.py:200-329`) defines:

| Surface | Protocol | InferenceEngine (v0.2.x) | HFEngine | Gap |
|---|---|---|---|---|
| `name` | `str` | `model_name` (different name) | ✓ | rename in callers |
| `tokenizer_id` | `str` (hash) | n/a | ✓ | — |

### Methods called outside Protocol

All of these are runtime-override kwargs that don't exist in the Protocol:

- `engine.score(prompts, continuations=..., system_prompt=..., context=..., progress=..., progress_desc=...)` — v0.3.2 PR #10 added these to `InferenceEngine` for engine reuse.
- `engine.generate(prompts, n_samples=..., max_new_tokens=..., system_prompt=..., context=..., decode=..., progress=..., progress_desc=...)` — same.

**Recommendation: (b) inline-rewrite**. The runtime overrides are
inherently **prompt-assembly** semantics (how to build the final
prompt string from `system_prompt + context + probe`), not engine
semantics. Move that assembly into `_pipeline` and pass the engine an
already-assembled string. This:

1. Keeps the Protocol unchanged (cleaner abstraction — the Engine
   doesn't need to know what a "system prompt" is).
2. Means `HFEngine.score(prompt, continuation)` is called per-probe
   in a Python `for` loop in `_pipeline`. Python overhead is the same
   as `InferenceEngine.score(list)` already does internally (see
   `lmdiff/engine.py:266` `for idx in _progress_iter(range(n), ...)`).
3. The progress bar wraps the pipeline-level loop, not the engine. This
   is conceptually nicer: the **pipeline** knows it's iterating over
   probes; the **engine** just scores one thing at a time.

The same recommendation handles the list-batching gap automatically.

## Audit 4 — BPB / cross-tokenizer

`bpb_from_ce(cross_entropy, n_tokens, text)` lives in
`lmdiff/tokenizer_utils.py:19`. It computes `(ce * n_tokens / log(2)) /
utf8_byte_count(text)` — pure math, no engine access.

`tokenizers_equivalent(tok_a, tok_b)` in the same module needs the
**tokenizer object** (calls `tok(text, add_special_tokens=False)`).
The Engine Protocol exposes only `tokenizer_id: str` (a hash of vocab
+ special tokens), not the tokenizer object.

**The fast path** (`engine_a.tokenizer_id == engine_b.tokenizer_id`)
covers most cases. The slow path is the L-011 "slow vs fast Llama
tokenizer have different vocab_size but produce identical canary-string
ids" edge case.

**Recommendation: (a) extend Protocol with one method.**

```python
# lmdiff/_engine.py
class Engine(Protocol):
    ...
    def tokenizers_equivalent_to(self, other: "Engine") -> bool:
        """True if this engine's tokenizer is substantively equivalent
        to ``other``'s. Default implementation compares ``tokenizer_id``
        for the fast path; subclasses owning a tokenizer object should
        override to also do canary-string comparison (see L-011)."""
        return self.tokenizer_id == other.tokenizer_id
```

`HFEngine` and `MockEngine` override to call the existing
`tokenizers_equivalent(self._tokenizer, other._tokenizer)` helper for
the slow-path canary check. Pipeline stays Protocol-clean.

This is the **only** Protocol extension this commit needs.

## Recommended cutover plan (for review)

Given the audit findings:

1. **`lmdiff/_pipeline.py` (new)** — `run_family_pipeline(base_engine,
   variant_engines, probe_set, *, max_new_tokens, task_overrides, seed,
   metrics, progress) -> GeoResult`. Owns:
   - Per-probe token-count precompute (uses `engine.token_count` —
     **Protocol extension #1**).
   - The engine cache + look-ahead-release loop (moved from
     `ChangeGeometry.analyze`).
   - The per-probe Python loop calling `engine.score(prompt,
     continuation)` once each. **Prompt assembly** (system_prompt +
     context + probe) happens here, not inside Engine.
   - BPB cross-tokenizer fallback via `engine.tokenizers_equivalent_to`
     (**Protocol extension #2**) + `bpb_from_ce`.
   - All the per-domain / share / overall computations (already pure
     functions in `lmdiff/geometry.py`; just import).

2. **`lmdiff/_engine.py`** — Protocol gains two methods:
   - `token_count(text: str) -> int` (default impl in helper, or
     subclass-required).
   - `tokenizers_equivalent_to(other: Engine) -> bool` (default impl
     compares `tokenizer_id`; HFEngine + MockEngine override for L-011
     canary check).

3. **`lmdiff/_api.py::compare()` / `family()`** — replaces the
   `ChangeGeometry(...).analyze(...)` block with
   `_pipeline.run_family_pipeline(base_engine, variant_engines,
   probe_set, ...)`. The engine-build / engine-close lifecycle stays
   in `_api`.

4. **`lmdiff/geometry.py::ChangeGeometry`** — kept as the v0.2.x
   library API (used by `run_family_experiment` and external scripts).
   Deprecation TBD; not in this commit.

5. **`lmdiff/experiments/family.py::run_family_experiment`** — wrap
   body with `warnings.warn(DeprecationWarning(...))` once per call.
   Internal `InferenceEngine` path unchanged.

6. **Tests** — update the 3 test files flagged in Audit 5; add the new
   pipeline-level test file `tests/unit/test_pipeline.py` (mock-engine
   only); add `tests/integration/test_calibration_regression.py` (GPU,
   reads `tests/fixtures/calibration_v032_baseline.json`).

7. **CHANGELOG + migration doc** — per Task 7 in the spec.

**Estimated diff size**: ~600–800 LoC added (`_pipeline.py` + tests +
2 Protocol methods on HFEngine/MockEngine), ~50 LoC removed
(simplified `_api.compare/family` body, simplified `ChangeGeometry`
won't actually shrink in this commit — it stays for the deprecated
path).

**Cutover scope**: BOTH `compare()` and `family()` in one commit. They
share the pipeline; splitting buys nothing. If the calibration
regression test fails for either, the whole commit doesn't ship.

## Calibration baseline — generate this on the GPU box, then commit

# Use to_dict() rather than save() — save() round-trips through to_json

with open("tests/fixtures/calibration_v032_baseline.json", "w") as f:
    payload = result.to_dict() if hasattr(result, "to_dict") else result
