# Migrating from v0.2.x to v0.3.0

v0.3.0 introduces a new top-level API (`lmdiff.compare`, `lmdiff.family`),
a frozen-and-validated `Config` dataclass with typed sub-specs, and a
backend-agnostic `Engine` Protocol. The pre-v0.3.0 surfaces continue to
work and now emit `DeprecationWarning`; the timeline below covers
when each is removed.

> **v0.4.0 backend cutover (no user action required).** As of v0.4.0,
> `lmdiff.compare()` and `lmdiff.family()` route through the public
> Engine Protocol (`HFEngine` by default) via the new
> `lmdiff/_pipeline.py` instead of the v0.2.x `InferenceEngine` /
> `ChangeGeometry` path. **Numeric output is byte-identical** to v0.3.2
> within 1e-6 per element on the calibration baseline — no migration
> steps required for users of `compare()` / `family()`. The deprecated
> `lmdiff.experiments.family.run_family_experiment` keeps the v0.2.x
> path internally and will continue to produce identical numbers
> through its own removal in v0.5.0.

## What changed

- **`lmdiff.Config`** is now the v0.3.0 dataclass at `lmdiff._config.Config`.
  It is `frozen=True`, hashable, validated at construction, and uses typed
  sub-specs (`DecodeSpec`, `AdapterSpec`, `QuantSpec`, `PruneSpec`,
  `KVCacheSpec`, `SteeringSpec`, `ICLExample`, `Message`).
- **`lmdiff.compare(base, variant, ...)`** and
  **`lmdiff.family(base, {name: variant, ...}, ...)`** are the new
  one-call entry points. They accept either model-id strings or `Config`
  instances, build engines internally, and clean up after themselves.
- **`Engine`** is a `runtime_checkable` Protocol. The canonical
  implementation is `HFEngine`; bring your own backend via `MinimalEngine`.
- **The pre-v0.3.0 `lmdiff.config.Config`** is the old dataclass with
  `decode: dict`, `dtype: str | None`, `ttt`, etc. It still works but
  emits `DeprecationWarning` on construction.
- **The pre-v0.3.0 `lmdiff.ModelDiff`** is the old wrapper. It still
  works but emits `DeprecationWarning` on construction;
  `ModelDiff.run(level="representation")` raises `NotImplementedError`
  pointing at v0.7.0 / Phase 5.

## Quick mapping table

| v0.2.x                                                              | v0.3.0                                                                 |
|---------------------------------------------------------------------|------------------------------------------------------------------------|
| `from lmdiff.config import Config`                                  | `from lmdiff import Config`                                            |
| `Config(model=..., decode={"strategy": "sample", "temperature": 0.7})` | `Config(model=..., decode=DecodeSpec(strategy="sample", temperature=0.7))` |
| `Config(model=..., dtype="bfloat16")`                               | (handled by Engine, not Config; see `HFEngine(dtype="bf16")`)          |
| `ModelDiff(a, b, prompts, n_samples=N).run()`                       | `lmdiff.compare(a, b, probes=prompts, n_probes=N)`                     |
| `ModelDiff(a, b, prompts).run(level="output")`                      | `lmdiff.compare(a, b, probes=prompts)` (default = output)              |
| `ModelDiff(a, b, prompts).run(level="representation")`              | `NotImplementedError` until v0.7.0 (Phase 5: representation metrics)   |
| `ChangeGeometry(base, {n: v}, prompts).analyze()`                   | `lmdiff.family(base, {n: v}, probes=prompts)`                          |

## Deprecation timeline

- **v0.3.0**: both surfaces active. `lmdiff.config.Config` and
  `lmdiff.ModelDiff` emit `DeprecationWarning` on construction. Old
  callers continue to work unchanged otherwise.
- **v0.4.0** (current): backend cutover for the new API path
  (`compare` / `family` route through `HFEngine`). Adds a new
  deprecation: `lmdiff.experiments.family.run_family_experiment` emits
  `DeprecationWarning` on call. The v0.2.x removals originally
  scheduled for v0.4.0 are pushed to v0.5.0 because
  `run_family_experiment` and `lmdiff.ModelDiff` still depend on
  `lmdiff.engine.InferenceEngine` and `lmdiff.geometry.ChangeGeometry`
  internally — removing those internals while the deprecated wrappers
  still exist would break their numeric guarantees mid-deprecation.
- **v0.5.0**: removal. `lmdiff.config` module deleted; `lmdiff.ModelDiff`,
  `run_family_experiment`, `lmdiff.engine.InferenceEngine`, and
  `lmdiff.geometry.ChangeGeometry` all removed.

If you need more time, pin `lmdiff-kit<0.5` in your `pyproject.toml`.

## Common gotchas

1. **`Config.decode` is no longer a dict.** Pass a `DecodeSpec`. Common
   recipe:
   ```python
   from lmdiff import DecodeSpec
   DecodeSpec(strategy="sample", temperature=0.7, top_p=0.95, max_new_tokens=64)
   ```
2. **`Config.dtype` is gone.** Precision is an Engine concern, not a
   Config concern. Pass `dtype="bf16"` to `HFEngine(...)` if needed; the
   default is `"bf16"`.
3. **`from lmdiff import Config` now resolves to the new class** even if
   you don't migrate everything else. `from lmdiff.config import Config`
   stays bound to the old class for the deprecation window.
4. **`compare()` and `family()` close engines they create themselves**
   in a `finally` block, so accidentally erroring mid-run no longer leaks
   GPU memory. If you pass `engine=...` as a template, you keep
   ownership.

## Configuration class deep-dive

`Config` is the **unit of comparison** in v0.3.0. Two `Config` instances
that differ in any field — model weights, adapter, quantization, system
prompt, decoding strategy, steering vector — represent two configurations
that lmdiff can compare. The shape is intentionally wide so the same
abstraction covers v0.7.0 representation work, v2.0 patching, and Phase 6
hosted-API backends.

```python
from lmdiff import (
    Config, AdapterSpec, QuantSpec, DecodeSpec, KVCacheSpec, ICLExample,
)

cfg = Config(
    model="meta-llama/Llama-2-7b-hf",
    adapter=AdapterSpec(type="lora", path="./my-lora", rank=8),
    quantization=QuantSpec(method="int4"),
    system_prompt="You are a careful expert.",
    decode=DecodeSpec(strategy="sample", temperature=0.7, max_new_tokens=64),
    icl_examples=(ICLExample(user="...", assistant="..."),),
    kv_cache_compression=KVCacheSpec(method="h2o", keep_ratio=0.5),
    name="llama-7b-lora-int4",
)
```

The class is `frozen=True`. Hashable when its fields are hashable
(arrays in `soft_prompts` / `SteeringSpec.vectors` are converted to
tuples for hash safety). Validated at `__post_init__` time — invalid
combinations like LoRA without `rank` raise `ValueError` immediately,
before any model load.

## Custom Engine integration

If you have a non-HF backend (vLLM, hosted API, custom inference rig),
implement the `Engine` Protocol via `MinimalEngine`:

```python
from lmdiff import MinimalEngine

class MyAPIEngine(MinimalEngine):
    def __init__(self, config):
        super().__init__(
            config,
            capabilities=frozenset({"score", "generate"}),
            n_layers=24, hidden_dim=4096,
        )
        self._client = MyAPIClient(config.model)

    def _score_impl(self, prompt, continuation):
        result = self._client.score(prompt, continuation)
        return result.logprobs, result.tokens

    def _generate_impl(self, prompt, max_new_tokens, temperature, top_p, seed):
        result = self._client.generate(
            prompt, max_tokens=max_new_tokens, temperature=temperature,
            top_p=top_p, seed=seed,
        )
        return result.text, result.tokens, result.logprobs
```

Pass an instance as `engine=` to `lmdiff.compare()` / `lmdiff.family()`.
The library treats it as a *template* and per-variant engines are built
via `template.with_config(cfg)`. You keep ownership of the template;
lmdiff does not close it.

Reserved capability names are listed in
`lmdiff._engine.RESERVED_CAPABILITIES` — pick from that list. Unknown
names raise `ValueError` at construction. New capability names need to
be added to the registry by a PR before backends can declare them.

## Reporting your migration experience

If you hit a v0.2.x → v0.3.0 paper cut not covered here — bad
deprecation warning, missing translation, broken script in the wild —
please file an issue at <https://github.com/MaiqiVerse/lmdiff/issues>
with:

- the v0.2.x snippet you started from,
- the v0.3.0 snippet you ended up with (or that errored),
- the lmdiff version (`pip show lmdiff-kit`),
- whether it surfaced as a `DeprecationWarning`, an `ImportError`, or
  silently produced wrong results.

The shim removal in **v0.4.0** is calibrated to your feedback during
the v0.3.x cycle.

## Reporting your results

v0.3.0 ships five renderers, all backed by the same `result.findings`:

| Channel | Method | Output |
|---|---|---|
| Terminal | `result.print()` | 5-layer ANSI to stdout |
| HTML | `result.to_html("report.html")` | Single self-contained file (~1 MB) |
| Markdown | `result.to_markdown("report.md")` | GitHub-flavored, with figure links |
| JSON | `result.save("result.json")` | Schema v5 (round-trips via `lmdiff.load_result`) |
| Figures | `result.figures(out_dir="figs/")` | 3 PNGs (drift+share, direction, change-size) |

All five draw narrative content from `result.findings` so the headline
text stays identical across channels.

## Where to look

- New `Config` and sub-specs: `lmdiff/_config.py`.
- New `Engine` protocol + `HFEngine`: `lmdiff/_engine.py`.
- New `compare()` / `family()`: `lmdiff/_api.py`.
- Findings registry: `lmdiff/_findings.py`.
- Renderers: `lmdiff/report/{terminal,html,markdown,json_report,figures}.py`.
