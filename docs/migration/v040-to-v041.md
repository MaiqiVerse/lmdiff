# Migrating from v0.4.0 to v0.4.1

> **TL;DR.** v0.4.1 ships two breaking changes: per-domain magnitude
> values shift (formula correction) and `share_per_domain[v][d]` may
> now be `None` for out-of-context domains. Existing v0.4.0 saves load
> with a `DeprecationWarning`; their numbers are preserved (not
> recomputed). Re-run your comparisons to get v0.4.1 numerics.

## What changed

### 1. Per-domain magnitude formula

v0.4.0 (and v0.3.2) used:

```
pdn[v][d] = sqrt(Σ_{i∈d} δ_i² / Σ_{i∈d} T_i)         ← v0.3.2 / v0.4.0
```

This formula was dimensionally inconsistent (units: `nats/token^1.5`)
and worked by accidentally suppressing long-context dominance through
the wrong mechanism. v0.4.1 uses the dimensionally clean Formula A:

```
pdn[v][d] = sqrt(mean_{i∈d, valid}(δ_i²))            ← v0.4.1
```

Units: `nats/token`. See `docs/methodology/normalization.md` for the
full derivation.

Empirical impact: `pdn_C / pdn_B = √T̄_d` for every (variant, domain)
pair — the new formula scales by `√T̄_d` relative to the old. Specific
numbers from the 4-variant calibration baseline are in
`docs/internal/v041_audit_4variant_check.py` output. The "biggest move"
domain is preserved for 6 of 7 demo variants; absolute share numbers
shift.

### 2. Per-probe measurement validity framework

Long-context probes (`longbench_2wikimqa`, ~9000 tokens) exceed
Llama-2-7B's 4096-token trained context. v0.4.1 detects these at the
per-(engine, probe) level and excludes them from per-domain
aggregation. They surface as `None` in `share_per_domain` and
`magnitudes_per_domain_normalized`, with a `domain_status` field
classifying each (variant, domain) pair as `full` / `partial` /
`variant_only` / `out_of_range`.

### 3. Schema bump v5 → v6

GeoResult JSON saves now use `schema_version: "6"`. Three new fields:

| field | type | meaning |
|---|---|---|
| `probe_validity` | `dict[probe_id → ProbeValidity]` | per-probe validity records across engines |
| `domain_status` | `dict[variant → dict[domain → str]]` | full / partial / variant_only / out_of_range |
| `variant_only_metrics` | `dict ∣ None` | reserved for v0.5.0+; always `None` in v0.4.1 |

### 4. Engine Protocol addition

`Engine.max_context_length() -> int | None`. HFEngine reads
`model.config.max_position_embeddings` with fallback to `n_positions`
then `max_seq_len`. Custom backends should override the
`MinimalEngine._max_context_impl` hook to expose their context limit.

## How to read the new fields

```python
import lmdiff
result = lmdiff.family(base="...", variants={...}, ...)

# Per-(variant, domain) status
for variant, statuses in result.domain_status.items():
    for domain, status in statuses.items():
        print(f"{variant} on {domain}: {status}")
# → yarn on long-context: variant_only
#   code on long-context: out_of_range
#   yarn on commonsense: full

# Share with None handling
for variant, row in result.share_per_domain.items():
    for domain, s in row.items():
        if s is None:
            print(f"  {variant}/{domain}: not measured (out_of_range or variant_only)")
        else:
            print(f"  {variant}/{domain}: {s * 100:.1f}%")

# Per-probe validity — which probes were skipped per engine
for probe_id, pv in result.probe_validity.items():
    if not pv.valid_for_all:
        print(f"{probe_id}: {pv.domain}")
        for eng_name, ev in pv.per_engine.items():
            print(f"  {eng_name}: valid={ev.is_valid} ({ev.reason}) T={ev.T_i}")
```

The shorthand `result.pdn` is an alias for
`result.magnitudes_per_domain_normalized`.

## How to interpret `partial` vs `out_of_range` in figures

`drift_share_dual.png` renders the validity status visually:

- **No hatch** — `full`. Numbers as computed.
- **`////` hatch + `*` suffix** — `partial`. The domain has some
  out-of-context probes; the displayed number is computed on the
  valid subset.
- **`xxxx` hatch + grey fill + `—`** — `out_of_range` or
  `variant_only`. Domain excluded from aggregation; no number to
  display.

## Loading v0.4.0 saves

v0.4.0 saves (schema v5) load with their original `share_per_domain`
and `magnitudes_per_domain_normalized` values preserved exactly —
"saved means saved" (Q9.8). A single `DeprecationWarning` fires on
load:

```
loaded GeoResult schema v5 (v0.3.2 - v0.4.0); values use the pre-v0.4.1
formula (sqrt(Σδ²/ΣT), dimensionally inconsistent — see L-033).
share_per_domain and magnitudes_per_domain_normalized preserved as
saved per v0.4.1 Q9.8 (saved means saved). Re-run with v0.4.1+ for
corrected numerics + per-probe validity records.
```

The loader synthesizes empty `probe_validity` and `"full"`-everywhere
`domain_status` so consumers can iterate the fields without `None`
checks, but the saved share/pdn numbers are NOT recomputed with the
v0.4.1 formula. To get v0.4.1 numerics on an old run, re-execute:

```python
import lmdiff
result = lmdiff.family(base="...", variants={...}, ...)  # produces v6 save
result.save("path/to/v041_baseline.json")
```

## FAQ

**Q: My yarn variant used to show 38% long-context share in v0.4.0 —
now `share_per_domain["yarn"]["long-context"]` is `None`. Why?**

Because Llama-2-7B's base context window is 4096 tokens, and 91/100
long-context probes in the demo exceed that. v0.4.0 included them in
the aggregate and produced an inflated "long-context share" number
that was actually measuring base-model failure, not variant
specialization. v0.4.1 flags long-context as `variant_only` for yarn
(yarn handles the prompts, base doesn't), so the base-vs-variant
comparison there has no meaningful signal. v0.5.0+ will surface
yarn-side metrics for variant-only domains via
`variant_only_metrics`.

**Q: Why is CodeLlama → code now 50% when v0.4.0 showed 32%?**

Because v0.4.0's √T̄ over-correction was suppressing in-context signal
across the board. With long-context probes correctly excluded from
the denominator AND the formula corrected to plain RMS, CodeLlama's
true specialty (code) gets its proper share weight. The narrative is
strengthened, not changed.

**Q: How do I reproduce v0.4.0 numbers?**

Install `lmdiff-kit==0.4.0`. v0.4.1 deliberately changes the numerics;
the v0.4.1 formula is methodologically grounded, the v0.4.0 numbers
were self-consistent-but-not-validated.

**Q: Do I need to do anything if I'm only consuming v0.4.1 saves
forward?**

No. The new fields are additive; old code that doesn't read
`probe_validity` / `domain_status` keeps working. The one place
existing code needs updating is any spot that does
`result.share_per_domain[v][d] * 100` (or equivalent) without a `None`
check — that will now raise on out-of-range domains. Defensive
update:

```python
def safe_pct(value):
    return None if value is None else f"{value * 100:.0f}%"
```

**Q: What's the GPU regen cost?**

About 2 hours total on a single RTX 5090 for the 4-variant + 7-variant
calibration fixtures. Scripts at `scripts/_regenerate_v041_4variant_fixture.py`
and `scripts/_regenerate_v041_7variant_fixture.py` (commit 8 of the
v0.4.1 PR).
